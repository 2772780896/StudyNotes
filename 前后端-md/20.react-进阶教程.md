## 第一章：Redux 状态管理全解

Redux 是 React 最常用的集中式状态管理工具，主要解决跨组件通信难题。

### 1.1 原生 Redux (深度理解)
虽然现在推荐使用 RTK，但理解原生 Redux 的原理是进阶的基础。

- **核心概念**:
  - **Store**: 存储数据的“仓库”。
  - **State**: 仓库中的数据状态。
  - **Action**: 一个普通的 JS 对象，描述“发生了什么”，必须有 `type` 属性。
  - **Reducer**: 一个纯函数，根据旧 State 和 Action，计算并返回新 State。
  - **Dispatch**: 派发 Action 的唯一手段。

- **工作流程**: 
  1. UI 发出指令 (`dispatch`)。
  2. Action 到达 Reducer。
  3. Reducer 根据 `action.type` 返回新对象。
  4. Store 更新，UI 重新渲染。

- **核心语法**:
  ```javascript
  // 1. 定义 Reducer
  function reducer(state = { count: 0 }, action) {
    switch (action.type) {
      case 'INCREMENT': return { count: state.count + 1 };
      default: return state;
    }
  }

  // 2. 创建 Store
  import { createStore } from 'redux';
  const store = createStore(reducer);

  // 3. 订阅与派发
  store.subscribe(() => console.log(store.getState()));
  store.dispatch({ type: 'INCREMENT' });
  ```

### 1.2 Redux Toolkit (RTK) - 现代标准
RTK 是官方推荐的工具集，解决了原生 Redux 配置繁琐、代码冗余的问题。

- **原理**: 引入了 **Slice (切片)** 的概念，将状态和逻辑模块化。内置了 **Immer** 库，允许我们在 Reducer 中以“直接修改”的方式编写逻辑（实际上生成了新对象）。

- **标准开发流程**:
  1. **安装**: `npm i @reduxjs/toolkit react-redux`
  2. **定义子模块 (Slice)**:
     ```javascript
     import { createSlice } from '@reduxjs/toolkit';

     const counterStore = createSlice({
       name: 'counter',
       initialState: { count: 0 },
       reducers: {
         increment(state, action) {
           // 注意：这里可以直接修改，RTK 底层会处理不可变性
           state.count += action.payload || 1;
         }
       }
     });

     export const { increment } = counterStore.actions;
     export default counterStore.reducer;
     ```
  3. **组合 Store**:
     ```javascript
     import { configureStore } from '@reduxjs/toolkit';
     import counterReducer from './modules/counterStore';

     const store = configureStore({
       reducer: { counter: counterReducer }
     });
     export default store;
     ```

### 1.3 React-Redux：连接组件与 Store
- **注入 Store**: 在入口文件 (`index.js`) 使用 `Provider` 包裹。
  ```jsx
  import { Provider } from 'react-redux';
  import store from './store';

  root.render(
    <Provider store={store}>
      <App />
    </Provider>
  );
  ```
- **组件内使用 (Hooks)**:
  - **useSelector**: 获取数据。`const count = useSelector(state => state.counter.count);`
  - **useDispatch**: 获取派发函数。`const dispatch = useDispatch();`
  - **语法**: `onClick={() => dispatch(increment(5))}`。

### 1.4 异步 Action (Thunk)
- **原理**: 默认 Reducer 只能处理同步。Thunk 允许我们在 Action Creator 中返回一个函数，在函数内执行异步操作后再派发。
- **语法流程**:
  ```javascript
  // 在 Slice 文件中定义
  export const fetchList = () => {
    return async (dispatch) => {
      const res = await axios.get('...');
      dispatch(setList(res.data)); // 异步完成后派发同步 Action
    };
  };
  ```

### 1.5 Zustand：现代极简状态管理 (推荐方案)
Redux 虽然强大但配置繁琐。**Zustand** 是目前 React 社区最火的替代方案。

- **原理**: 基于闭包和发布订阅模式。它不需要 `Provider` 包裹根组件，通过创建一个自定义 Hook 来直接管理状态，消除了“模板代码”。

- **标准开发流程**:
  1. **安装**: `npm i zustand`
  2. **创建 Store (`store/useCounterStore.js`)**:
     ```javascript
     import { create } from 'zustand';

     const useCounterStore = create((set) => ({
       count: 0,
       // 同步修改
       inc: () => set((state) => ({ count: state.count + 1 })),
       // 异步修改 (直接支持，无需 Thunk)
       asyncInc: async () => {
         await new Promise(resolve => setTimeout(resolve, 1000));
         set((state) => ({ count: state.count + 1 }));
       }
     }));
     export default useCounterStore;
     ```
  3. **组件内使用**:
     ```jsx
     import useCounterStore from './store/useCounterStore';

     function Counter() {
       const { count, inc, asyncInc } = useCounterStore();
       return <button onClick={inc}>{count}</button>;
     }
     ```

- **进阶：持久化中间件 (persist)**:
  - **场景**: 自动将状态同步到 `localStorage`，刷新不丢失。
  - **语法**:
    ```javascript
    import { persist } from 'zustand/middleware';
    const useStore = create(
      persist(
        (set) => ({ ... }),
        { name: 'storage-key' } // 存储在本地的键名
      )
    );
    ```

## 第二章：React Router 路由深度指南

React Router 允许我们根据 URL 的变化渲染不同的组件。

### 2.1 路由开发模式
在实际开发中，有两种组织路由代码的方式：

#### **(1) 统合模式 (简易版)**
- **做法**: 直接在 `index.js` 中配置路由并渲染。
- **场景**: 极小型项目或演示 demo。

#### **(2) 模块化模式 (推荐方案)**
- **原理**: 遵循“单一职责”原则，将页面、路由配置、入口文件解耦。
- **目录流程**:
  1. **src/page/**: 存放页面组件（如 `page/Home/index.js`）。
  2. **src/router/index.js**: 统一导入页面，配置并导出 `router` 实例。
  3. **src/index.js**: 导入 `router` 实例，通过 `<RouterProvider router={router} />` 挂载。

### 2.2 路由模式深度对比
- **BrowserRouter (推荐)**: 
  - **原理**: 基于 HTML5 的 History API。
  - **表现**: URL 正常（如 `/home`）。
  - **注意**: 刷新页面时浏览器会向服务器请求该路径，需要后端配置将所有路径指向 `index.html`。
- **HashRouter**:
  - **原理**: 基于 `window.location.hash`。
  - **表现**: 带有 `/#/`（如 `/#/main/#/page1`）。
  - **注意**: 改变哈希值不会触发浏览器向服务器发请求，因此不需要后端特殊配置。

### 2.3 导航与跳转细节
- **Link 标签**:
  - **语法**: `<Link to="/path">跳转</Link>` (绝对路径) 或 `<Link to="path">跳转</Link>` (相对路径)。
  - **原理**: 拦截点击事件，仅修改地址栏并触发 React 内部重绘。
  - **为什么不用 <a>?**: `<a>` 会引起整页刷新，导致 React 内存中的所有状态（State/Redux）全部丢失。
- **useNavigate (编程式导航)**:
  - **基本跳转**: `navigate('/path')`。
  - **状态传递**: `navigate('/path', { state: { from: 'home' } })`。

### 2.4 参数传递与获取 (三套方案)
| 方案 | 语法 (传递) | 语法 (接收) | 场景 |
| :--- | :--- | :--- | :--- |
| **SearchParams** | `navigate('/user?id=1')` | `const [params] = useSearchParams(); params.get('id')` | 过滤、分页 |
| **Params** | `navigate('/user/1')` | `const { id } = useParams();` | 详情页 (需在路由表配置 `:id`) |
| **State** | `navigate('/u', {state:{id:1}})` | `const location = useLocation(); location.state.id` | 隐藏传参 |

### 2.5 嵌套路由与特殊配置
- **二级路由 (children)**:
  - **配置**: 在父路由下增加 `children` 数组。
  - **渲染**: 在父组件中使用 `<Outlet />` 作为子路由的占位符。
- **默认渲染 (index)**:
  - **语法**: `{ index: true, element: <Home /> }`。
  - **效果**: 访问父路径时，默认在 `Outlet` 位置渲染该组件。
- **404 路由**:
  - **语法**: `{ path: '*', element: <NotFound /> }`。
  - **效果**: 当 URL 无法匹配所有已知路径时，显示该组件。

### 2.6 路由懒加载 (Performance)
- **原理**: 分包 (Code Splitting)。只有当用户访问该路径时，才去服务器下载对应的 JS 文件。
- **流程**:
  1. 使用 `lazy` 导入组件。
  2. 用 `Suspense` 包裹组件，并设置 `fallback`（加载中状态）。
  ```javascript
  const Home = lazy(() => import('../pages/Home'));
  // 路由配置
  { path: '/', element: <Suspense fallback="Loading..."><Home /></Suspense> }
  ```

## 第三章：Ant Design 与 UI 组件库实践

Ant Design (antd) 是业界领先的 React UI 组件库。

### 3.1 核心组件用法
- **安装**: `npm i antd`
- **Button / Card**: 基础展示组件。
- **Form (表单处理)**:
  - **原理**: 集中管理表单状态与校验逻辑。
  - **关键语法**: 
    - `onFinish`: 校验通过后触发的回调，直接接收数据对象。
    - `initialValues`: 设置初始值。
    - `form.setFieldsValue(data)`: 用于**数据回填**（例如编辑回显）。
- **Upload (文件上传)**:
  - **流程**: 用户选文件 -> `onChange` 状态变化 -> 自动或手动上传到 `action` 接口。
  - **数据格式**: 通常为 `multipart/form-data`。

## 第四章：项目架构封装与优化

### 4.1 Axios 深度封装
- **原理**: 统一请求的基础配置，并通过拦截器处理通用逻辑。
- **流程**:
  1. **请求拦截器**: 在请求发出前，自动从本地获取 Token 并注入到 Header。
  2. **响应拦截器**: 统一处理状态码（如 401 时跳转登录页），简化返回的数据结构。
- **语法示例**:
  ```javascript
  const request = axios.create({ baseURL: '...', timeout: 5000 });

  request.interceptors.request.use(config => {
    const token = getToken();
    if (token) config.headers.Authorization = `Bearer ${token}`;
    return config;
  });

  request.interceptors.response.use(
    res => res.data, // 简化数据层级
    error => {
      if (error.response?.status === 401) navigate('/login');
      return Promise.reject(error);
    }
  );
  ```

### 4.2 API 模块化封装
- **原理**: 将所有的接口请求抽离到 `apis/` 目录下，组件内只管调用函数。
- **流程**: 组件触发请求 -> 调用 API 函数 -> API 函数调用封装好的 Request -> 返回数据。

### 4.3 跨域与开发代理

#### **(1) 同源策略与跨域**
- **同源定义**: 协议 + 域名 + 端口 三者完全一致才算同源。
- **典型场景**: 前端 `localhost:5173` 请求后端 `localhost:3000` → 端口不同 → 跨域。
- **浏览器行为**: 拦截跨域响应（除非后端返回 `Access-Control-Allow-Origin` 响应头）。
- **解决方案对比**:

| 方案 | 原理 | 适用场景 |
|------|------|----------|
| 后端 CORS 中间件 | 后端允许跨域请求 | 开发 + 生产 |
| Vite Proxy | 开发服务器转发，绕过浏览器限制 | 仅开发 |
| Nginx 反向代理 | 服务器层转发请求 | 生产 |

#### **(2) fetch 与 axios 对比**
| 特性 | fetch (浏览器原生) | axios (npm 库) |
|------|----------------|----------------|
| JSON 解析 | 手动 `.json()` | 自动 |
| 错误处理 | 404/500 不抛错 | 自动抛错 |
| baseURL | 无 | 有 |
| 拦截器 | 无 | 有（统一加 Token 等） |
| 适用场景 | 小型项目够用 | 大型项目推荐 |

#### **(3) Vite Proxy 原理与配置**
- **原理**: Vite 开发服务器（Node.js）不受同源策略限制，可作为中间人转发请求。
- **数据流**:
  ```
  浏览器 → fetch('/api/upload') → Vite(:5173) → 转发 → Express(:3000)
                                                           ↓
  浏览器 ← 响应 ← Vite(:5173) ← 原样返回 ← Express(:3000)
  ```
- **配置**:
  ```javascript
  // vite.config.js
  export default defineConfig({
    server: {
      proxy: {
        '/api': {
          target: 'http://localhost:3000',  // 转发目标
          changeOrigin: true,               // 修改请求头 Host 为 target 地址
        }
      }
    }
  })
  ```
- **changeOrigin**: 转发时将请求头的 `Host` 从 `localhost:5173` 改为 `localhost:3000`。某些后端会校验 Host，设为 `true` 最安全，几乎永远写 `true`。
- **局限**: 只在开发环境生效，打包后无 Vite 服务器，生产由 Nginx 处理。

### 4.4 高阶组件 (HOC) 与逻辑封装
高阶组件是 React 中复用组件逻辑的高级技巧。

#### **(1) 性能优化：React.memo**
- **原理**: 如果一个组件在相同 `props` 下渲染结果一致，`React.memo` 会缓存渲染结果，跳过重绘。
- **语法**:
  ```jsx
  const MyComponent = React.memo(({ data }) => {
    return <div>{data}</div>;
  });
  ```

#### **(2) 权限控制：AuthRoute 完整实现**
- **场景**: 实现“只有登录后才能访问某些页面”的逻辑。
- **编写组件 (`components/AuthRoute.js`)**:
  ```jsx
  import { Navigate } from 'react-router-dom';
  import { getToken } from '@/utils'; // 假设已封装获取 token 的方法

  function AuthRoute({ children }) {
    const token = getToken();
    if (token) {
      // 存在 token，正常渲染被包裹的子组件
      return <>{children}</>;
    } else {
      // 不存在 token，重定向到登录页，并使用 replace 替换历史记录
      return <Navigate to="/login" replace />;
    }
  }
  export { AuthRoute };
  ```
- **在路由中使用 (`router/index.js`)**:
  ```jsx
  const router = createBrowserRouter([
    {
      path: '/layout',
      // 通过组件嵌套写法，将受保护的页面作为 children 传入
      element: <AuthRoute> <Layout /> </AuthRoute>,
      children: [...]
    }
  ]);
  ```

### 4.5 Token 持久化深度处理
- **痛点**: Redux 状态在刷新后会重置。
- **方案**: 结合 `localStorage` 确保存储。
- **流程**: 
  1. **存储**: 登录成功派发 Action 时，同步执行 `localStorage.setItem('token_key', token)`。
  2. **初始化**: 在 Slice 的 `initialState` 中读取：
     ```javascript
     initialState: {
       token: localStorage.getItem('token_key') || ''
     }
     ```

## 第五章：工程化配置全解

### 5.1 路径别名 (Alias) 配置指南
为了避免 `import ../../../utils` 这种混乱路径，我们通常配置 `@` 指向 `src` 目录。

#### **第一步：安装 craco (针对 CRA 项目)**
`npm i @craco/craco -D`

#### **第二步：创建 `craco.config.js`**
在项目根目录创建该文件，配置 Webpack 别名：
```javascript
const path = require('path');
module.exports = {
  webpack: {
    alias: {
      // 配置 @ 符号指向 src 目录
      '@': path.resolve(__dirname, 'src')
    }
  }
};
```

#### **第三步：修改 `package.json`**
将启动命令从 `react-scripts` 改为 `craco`：
```json
"scripts": {
  "start": "craco start",
  "build": "craco build"
}
```

#### **第四步：配置路径提示 (`jsconfig.json`)**
在根目录创建 `jsconfig.json`，让 VSCode 识别别名并提供路径补全：
```json
{
  "compilerOptions": {
    "baseUrl": "./",
    "paths": {
      "@/*": ["src/*"]
    }
  }
}
```

### 5.2 CDN 排除与注入
- **原理**: 生产环境减少包体积。
- **craco 配置**:
  ```javascript
  // craco.config.js
  const { whenProd } = require('@craco/craco');
  module.exports = {
    webpack: {
      configure: (webpackConfig) => {
        whenProd(() => {
          // 告诉 Webpack 这些包不参与打包
          webpackConfig.externals = {
            react: 'React',
            'react-dom': 'ReactDOM'
          };
        });
        return webpackConfig;
      }
    }
  };
  ```
- **HTML 注入**: 在 `public/index.html` 中手动引入对应的 CDN `<script>` 标签。

## 第六章：类组件与 React + TypeScript

### 6.1 类组件 (Class Component)
虽然现代开发主推 Hooks，但理解类组件有助于维护旧项目。
- **核心**: 继承 `React.Component`。
- **状态**: `this.state` 定义，`this.setState` 修改（也是异步的）。
- **生命周期**:
  - `componentDidMount`: 相当于 `useEffect(..., [])`，挂载后执行。
  - `componentWillUnmount`: 相当于 Effect 的清理函数。

### 6.2 React + TypeScript (进阶必备)
- **原理**: 静态类型检查，显著减少 Bug。
- **核心语法**:
  - **Props 约束**: 使用 `interface` 定义。
  - **Hooks 约束**: 
    - `useState<number>(0)`
    - `useRef<HTMLDivElement>(null)`
  - **事件类型**: `React.ChangeEvent<HTMLInputElement>`。
